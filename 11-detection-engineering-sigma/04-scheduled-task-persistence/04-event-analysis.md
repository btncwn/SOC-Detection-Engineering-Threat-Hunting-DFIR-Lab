# Event Analysis – Suspicious Scheduled Task Creation

## Overview

After validating the Sigma detection against BOTSv3 telemetry, a single high-confidence event was identified that matched all detection conditions.

This document demonstrates how a SOC analyst would investigate and interpret the event.

---

# Event Summary

The event was generated by Sysmon Event ID 1.

```text
Event ID: 1
Source: Sysmon
Host: FYODOR-L
```

Sysmon Event ID 1 represents:

```text
Process Creation
```

The primary investigative question is:

```text
Who launched what process and why?
```

---

# Step 1 – Identify the Executed Process

The first field analysts should examine is:

```text
Image
```

Observed value:

```text
C:\Windows\System32\schtasks.exe
```

This indicates that the Windows Task Scheduler command-line utility was executed.

---

# Step 2 – Examine the Command Line

The next field is:

```text
CommandLine
```

Observed value:

```text
schtasks.exe /Create /F /RU system /SC DAILY /ST 18:45 /TN Updater
```

Breaking the command apart:

| Parameter   | Meaning                       |
| ----------- | ----------------------------- |
| /Create     | Create a scheduled task       |
| /F          | Force overwrite existing task |
| /RU system  | Execute as SYSTEM             |
| /SC DAILY   | Run daily                     |
| /ST 18:45   | Execute at 18:45              |
| /TN Updater | Task name                     |

This immediately indicates task creation activity.

---

# Step 3 – Identify the Payload

The scheduled task was configured to execute:

```text
powershell.exe
```

with the following indicators:

```text
-NonI
-W hidden
IEX(...)
```

### Why This Is Suspicious

#### -NonI

```text
Non Interactive Mode
```

The command executes without user interaction.

#### -W hidden

```text
Hidden Window
```

The PowerShell console is hidden from the user.

#### IEX

```powershell
Invoke-Expression
```

This executes dynamically generated PowerShell code.

Attackers frequently use IEX to execute downloaded, encoded, or obfuscated payloads.

---

# Step 4 – Identify the Parent Process

The most important field for this investigation is:

```text
ParentImage
```

Observed value:

```text
C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe
```

Process relationship:

```text
PowerShell
      ↓
schtasks.exe
```

This parent-child relationship formed the basis of the Sigma detection.

---

# Step 5 – Identify the User Context

Observed value:

```text
AzureAD\FyodorMalteskesko
```

The scheduled task was created by a user account.

However, the task was configured to execute as:

```text
SYSTEM
```

This results in a privilege transition from a user context to the highest local privilege level.

---

The event satisfied all Sigma detection conditions.

| Detection Condition             | Result |
| ------------------------------- | ------ |
| Parent Process = powershell.exe | PASS   |
| Child Process = schtasks.exe    | PASS   |

Detection Status:

```text
VALIDATED
```

# Analyst Assessment

Several characteristics make this event suspicious:

* PowerShell spawned schtasks.exe
* A scheduled task was created
* The task runs as SYSTEM
* Hidden PowerShell execution was configured
* Invoke-Expression (IEX) was present
* Encoded PowerShell content was observed

Taken together, these indicators strongly suggest malicious persistence activity rather than routine administrative task creation.

This event represents a high-confidence detection and demonstrates how process creation telemetry can be leveraged to identify attacker persistence mechanisms through Sigma-based detection engineering.
